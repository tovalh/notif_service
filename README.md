# notif_service

[English](#english) · [Español](#espanol)

<a id="english"></a>

A small real-time notification service written in Go. It watches one or more
MySQL databases for new rows in `usu_notificacion` and pushes them over
WebSocket to the right user, so the browser updates the moment something
happens instead of waiting for a refresh.

Polling is the source of truth (it works even if nobody tells the service
anything), and an optional `POST /notify` endpoint lets the backend push
instantly for lower latency. Both paths end up in the same hub.

## Features

- **WebSocket delivery per user.** A concurrent hub keyed by
  `(idgen_empresa, idusu_usuario)` fans each event out to that user's open
  connections. Isolation lives in the hub, not in every message.
- **Polling as the source of truth.** One goroutine per database, each with
  its own cursor (`MAX(idusu_notificacion)`), so a restart never re-pushes old
  notifications and the service works without any push wiring.
- **Optional instant push.** `POST /notify` lets the backend hand off an event
  the moment it's created, dropping latency from the poll interval to ~instant.
- **Signed-token auth.** Connections are authenticated with an HMAC-SHA256
  token (same `firma.payload` scheme as the PHP/JS backend), validated in
  constant time and checked for expiry — no shared session needed.
- **Multi-database.** Fill `DB_DSN_2..4` and each new DSN spins up its own
  poller against the same hub; no code changes.

## How it works

```
[backend]  INSERT into usu_notificacion
     |         └── (optional) POST /notify  ── instant path
     v
[MySQL]  usu_notificacion
     ^
     |  poller reads new ids every POLL_SECONDS   ── source of truth
     v
[hub]  (empresa, usuario) -> [ *websocket.Conn ]
     v
[browser]  ws.onmessage -> reload the bell via the existing REST API
```

The message the browser receives is a small ping, not the full payload:

```json
{ "tipo": "notif", "id": 123 }
```

so the front just re-fetches its notifications endpoint — the service never has
to replicate formatting or unread-count logic.

## Running locally

Requires a MySQL reachable via `DB_DSN` with a `usu_notificacion` table
(`idusu_notificacion`, `idgen_empresa`, `idusu_usuario`, `habilitado`).

```bash
cp .env.example .env      # set DB_DSN and WS_SECRET
go run .
# -> listening on http://localhost:8081/health
```

Quick smoke test:

```bash
# health
curl localhost:8081/health          # {"status":"ok"}

# connect a browser/client to the WS with a signed token, then insert a row:
#   INSERT INTO usu_notificacion (idgen_empresa, idusu_usuario, habilitado) ...
# within POLL_SECONDS the client receives {"tipo":"notif","id":<n>}
```

The token is generated by the backend (it holds `WS_SECRET`); the browser
connects to `GET /ws?token=<token>`.

## Configuration

All settings come from the environment (see `.env.example`):

| Variable       | Default | Meaning                                              |
| -------------- | ------- | ---------------------------------------------------- |
| `DB_DSN`       | —       | Primary MySQL DSN: `user:pass@tcp(host:port)/db`     |
| `DB_DSN_2..4`  | empty   | Extra databases; each one starts its own poller      |
| `WS_ADDR`      | `:8081` | Listen address for the HTTP/WebSocket server         |
| `WS_SECRET`    | —       | Shared secret used to validate tokens (== backend)   |
| `POLL_SECONDS` | `5`     | How often each poller checks its database            |

`WS_SECRET` **must be identical** to the secret the backend uses to sign
tokens.

## Endpoints

| Method | Path       | Purpose                                                    |
| ------ | ---------- | ---------------------------------------------------------- |
| `GET`  | `/health`  | Liveness probe → `{"status":"ok"}`                         |
| `GET`  | `/ws`      | WebSocket handshake; `?token=<token>` required             |
| `POST` | `/notify`  | Optional instant push from the backend (low-latency path)  |

## Token format

```
token = base64url( HMAC_SHA256(payloadJSON, WS_SECRET) ) + "." + base64url(payloadJSON)
payloadJSON = {"idusu_usuario": 42, "idgen_empresa": 5, "exp": <unix_ts>}
```

`base64url` uses the `-_` alphabet with no padding, matching the PHP/JS
`ApiConnector`. The service splits on `.`, re-signs the payload with
`WS_SECRET`, compares in constant time and rejects the token if the signature
mismatches or `exp` has passed.

## Layout

```
main.go       entrypoint: load config, open DB pollers, start HTTP server
config.go     env config loading (DSNs, secret, address, interval)
poller.go     per-DB polling loop with a cursor -> pushes new notifications
hub.go        concurrent (empresa, usuario) -> connections registry + fan-out
ws.go         /ws handshake: validate token, register/unregister the connection
auth.go       HMAC-SHA256 token parsing/validation (PHP/JS-compatible scheme)
notify.go     /notify handler for the optional instant-push path
client.go     per-connection helpers
```

## Español

Un pequeño servicio de notificaciones en tiempo real escrito en Go. Vigila una o
más bases de datos MySQL en busca de filas nuevas en `usu_notificacion` y las
empuja por WebSocket al usuario correcto, para que el navegador se actualice
apenas pasa algo, en vez de esperar a un refresh.

El polling es la fuente de verdad (funciona aunque nadie le avise nada al
servicio), y un endpoint opcional `POST /notify` permite que el backend empuje
al instante para bajar la latencia. Ambos caminos terminan en el mismo hub.

### Características

- **Entrega por WebSocket por usuario.** Un hub concurrente indexado por
  `(idgen_empresa, idusu_usuario)` reparte cada evento a las conexiones abiertas
  de ese usuario. El aislamiento vive en el hub, no en cada mensaje.
- **Polling como fuente de verdad.** Una goroutine por base de datos, cada una
  con su propio cursor (`MAX(idusu_notificacion)`), así un reinicio nunca
  re-empuja notificaciones viejas y el servicio funciona sin ningún cableado de
  push.
- **Push instantáneo opcional.** `POST /notify` permite que el backend entregue
  un evento apenas se crea, bajando la latencia del intervalo de polling a
  ~instantáneo.
- **Auth con token firmado.** Las conexiones se autentican con un token
  HMAC-SHA256 (mismo esquema `firma.payload` que el backend PHP/JS), validado en
  tiempo constante y con chequeo de expiración — sin compartir sesión.
- **Multi-base.** Rellená `DB_DSN_2..4` y cada DSN nuevo levanta su propio
  poller contra el mismo hub; sin cambios de código.

### Cómo funciona

```
[backend]  INSERT en usu_notificacion
     |         └── (opcional) POST /notify  ── vía instantánea
     v
[MySQL]  usu_notificacion
     ^
     |  el poller lee ids nuevos cada POLL_SECONDS   ── fuente de verdad
     v
[hub]  (empresa, usuario) -> [ *websocket.Conn ]
     v
[navegador]  ws.onmessage -> recarga la campanita vía la REST API existente
```

El mensaje que recibe el navegador es un ping chico, no el payload completo:

```json
{ "tipo": "notif", "id": 123 }
```

así el front solo vuelve a pedir su endpoint de notificaciones — el servicio
nunca tiene que replicar el formato ni la lógica de conteo de no-leídas.

### Formato del token

```
token = base64url( HMAC_SHA256(payloadJSON, WS_SECRET) ) + "." + base64url(payloadJSON)
payloadJSON = {"idusu_usuario": 42, "idgen_empresa": 5, "exp": <unix_ts>}
```
